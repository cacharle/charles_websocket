The WebSocket Protocol

RFC 6455

The Problem

HTTP is request-response.

For real-time data, workarounds were needed:

PollingWasteful — most responses are empty
Long-pollingKeeps connections hanging
SSEServer → client only

We need bidirectional messaging.

Opening Handshake

Starts as a normal HTTP request, then upgrades.

GET /chat HTTP/1.1
Host: server.example.com
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Key: dGhlIHNhbXBsZSBub25jZQ==
Sec-WebSocket-Version: 13
HTTP/1.1 101 Switching Protocols
Upgrade: websocket
Connection: Upgrade
Sec-WebSocket-Accept: s3pPLMBiTxaQ9kYGzzhZRbK+xOo=

After 101, the TCP connection is no longer HTTP — it carries WebSocket frames.

permessage-deflate

Compression extension, negotiated during the handshake:

Sec-WebSocket-Extensions: permessage-deflate;
    client_max_window_bits=15; server_max_window_bits=15
  • Compresses payloads using DEFLATE
  • Signaled by RSV1 = 1 in the frame header
  • Only data frames — not control frames

WSS — WebSocket over TLS

TLS wraps the TCP connection — the WebSocket protocol itself is unchanged.

Why the strange GUID and masking key?

Without them, an attacker can craft WebSocket payloads that look like HTTP to a transparent proxy, poisoning its cache.

  • Masking — makes the wire bytes unpredictable, so proxies can't interpret them as HTTP
  • GUID — proves that the server handles the WebSocket protocol
